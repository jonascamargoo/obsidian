# Fundamentals and Architecture of Semantic Caching

In this lesson, we explore *why* semantic caching is crucial for AI Agents (which are "token hungry") and analyze a reference architecture based on the Walmart use case.

## 📉 The Cost of Intelligence
There is a real tension between quality and cost/latency:
* Complex reasoning models (e.g., GPT-5, Claude) are more expensive and slower.
* **AI Agents:** Perform loops of planning, reflection, and multiple calls. A single task can cost significantly (e.g., USD $6.80) and take a long time.
* **RAG:** Increases prompt size (context), raising execution costs.

## ⚙️ The Semantic Caching Workflow

Unlike exact caching (which fails with natural language variations), semantic caching follows this flow:

1.  **Embed:** Convert the user question into a vector.
2.  **Vector Search:** Compare the query similarity with entries in the cache.
3.  **Classify (Threshold):**
    * *If distance is small (high similarity):* **Cache HIT** (Return stored answer).
    * *If distance is large:* **Cache MISS** (Call the RAG/LLM system).
4.  **Update:** If there is a Miss, the new response generated by the LLM is vectorized and stored in the cache for future use.

> [!WARNING] The Risk
> Semantic caching increases **Recall** (finds more matches) but introduces the risk of **False Positives** (delivering a wrong answer for a question that looked similar but wasn't).

## 📊 Effectiveness Metrics
To measure success in production, we focus on four main metrics. Ideally visualized via a Confusion Matrix.



![[Pasted image 20251125180425.png]]


1.  **Cache Hit Rate:** Frequency of cache usage. Directly impacts **cost savings**.
2.  **Precision:** How correct the answers are when the cache is hit.
3.  **Recall:** The ability to cover repeated questions (avoiding unnecessary misses).
4.  **Latency:** Total response time (cache must be extremely fast to avoid being a bottleneck).

### Enhancement Techniques
* **Threshold Tuning:** Calibrating vector distance sensitivity.
* **Rerankers:** Using Cross-Encoders or an "LLM Judge" to verify similarity.
* **Fuzzy Matching:** Handling typos before vectorization (saves compute).

## 🏢 Case Study: "waLLMartCache"
Walmart published a semantic caching architecture that achieved nearly **90% accuracy**.

### Reference Architecture
1.  **Load Balancer:** For horizontal scaling.
2.  **Dual-tiered Storage:**
    * **L1 (Vector DB):** Pure similarity search (returns IDs).
    * **L2 (In-Memory/Redis):** Data and metadata storage (Fast lookup by ID).
3.  **Multi-tenancy:** Support for multiple teams/apps on the same infrastructure.

![[Pasted image 20251125180515.png]]

### The Decision Engine
Before searching the cache, the system analyzes the query to decide if the cache should be **bypassed**.
* **Code Detection:** If the question involves code (Python, Java), skip cache.
* **Temporal Context:** If the question is time-sensitive (e.g., "price today"), skip cache.

> [!TIP] Architectural Insight
> Preliminary filters (like the Decision Engine) are essential to prevent the cache from returning obsolete or technically incorrect information in specific domains.

## 🤖 Agent Application (LangGraph)
In the final course project, the cache will be integrated into a LangGraph workflow:
1.  User sends a complex query.
2.  Agent **decomposes** the query into smaller sub-questions.
3.  Checks the cache for **each sub-question**.
4.  Synthesizes cached answers with new LLM calls only when necessary.
